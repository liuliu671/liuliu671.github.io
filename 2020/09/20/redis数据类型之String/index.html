String存储原理：
redis是一个KV的数据库，把这种KV的数据叫做字典dict,用hashtable实现
每个键值对都会有一个 dictEntry里面指向了 key 和 value 的指针。next 指向下一个 dictEntry。

key是存在SDS的，value是存在redisObject里的，然后再由redisObject指定具体的数 据结构，比如value也是一个字符串，那么又会指向一个SDS

可以使用type命令来查看对外的数据类型

用object encoding 命令来查看内部实际编码

SDS是指简单动态字符串（simple dynamic string），redis中字符串的实现。
字符串类型的内部编码有三种（redis5.5）
1.int，存储8个字节的长整型
2.embstr，是存储小于44个字节的字符串
3.raw，存储大于44个字节的字符串

用不同的编码实现同一种数据类型的目的是为了节省内存！！

embstr和raw的区别：

  1. embstr存储redisObject和SDS时，是连续分配内存空间的，只分配一次内存空间；而 raw 需要分配两次内存空间
  2. embstr的编码是只读的，当ebmstr的sds进行修改，就会先转换成raw，再进行修改，无论是否达到44个字节。

遵守一个原则：只会从小内存编码长度变为大的，不会从大的变小的，是不会还原的，但不包括重新赋值set。

redis是用C语言实现的，C语言是没有字符串类型的，它只能用字符数组char[]来实现字符串。
char[]缺点：
1.内存，数组是必须先分配内存空间，如果分配的空间不够，可能会造成内存溢出；
2.长度，获取字符数组的长度很复杂，需要遍历来获取，O(n) 时间由字符数组的长度来决定；
3.长度变化，若字符数组的长度发生了变化，要重新分配内存；
4.’\0 ‘对于二进制不安全,不安全的原因在于C语言中是根据空字符‘\0’来判断字符串是否已结尾，这样就不能保存例如图片、视频这样的二进制数据。

Redis使用SDS的好处：
1.不需要担心内存溢出的问题，长度发生变化，会进行扩容
2.SDS中专门使用len属性来保存字符串的长度，就不需要去遍历，获取字符串长度的时间复杂度为O(1)，是固定的
3.无论SDS存储内容变大变小，不是直接进行内存的增加和释放，它有空间预分配和惰性空间释放（可减少内存分配的次数）
4.根据len字符长度来判断字符串是否结束

String 的应用场景：
1.缓存！一些热点数据、全页面的静态页面的缓存，可以提升热点数据的访问速度
2.分布式的session！
3.set NX EX 可用于实现分布式锁
4.incr 可做全局ID的实现，分库分表的场景里
5.incr 基于原子递增的，不管有多少用户同时访问，都是一个个增加的，可用来做计数器，比如文章的阅读量、微博的点赞数
6.incr 可用于一个限流的业务场景，比如用访问者的信息+ip作为一个id，每访问一次就+1，到达访问限制，就不能进行访问了。
7.位操作，可用于大数据的统计
